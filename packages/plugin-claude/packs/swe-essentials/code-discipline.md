# Code Discipline

## Scope
- **Change only what was asked** — no unsolicited refactors, docstrings, renames, or "improvements".
- **No speculative code** — no error handling for impossible scenarios, no abstractions for a single use case.
- **No future-proofing comments** — no `// TODO: add X later`, no placeholder stubs, no deferred work markers.

## Consistency
- **Read before writing** — understand existing patterns before adding new ones. Match them.
- **Follow conventions, not preferences** — use the project's naming, structure, and style even if you'd do it differently.

## Cleanliness
- **Delete completely** — no commented-out code, no `_unused` suffixes, no `// removed` markers.
- **Comments explain WHY, not WHAT** — the code shows what. Comments explain intent, trade-offs, or non-obvious constraints.
- **No AI attribution** — no "Generated by", no authorship comments.

## Clarity
- **Explicit over clever** — readable logic beats a clever one-liner.
- **Early returns** — reduce nesting with guard clauses rather than deep if/else chains.
- **Named intermediates** — extract a well-named variable rather than a deeply nested expression.

## Trust
- **Validate at boundaries only** — trust internal code, types, and framework guarantees. Don't defensively check what the type system already enforces.
